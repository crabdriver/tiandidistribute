# 2026-03-28 浏览器托管会话手工验证

## 目标

验证浏览器平台已经从“每次手动准备远程调试 Chrome”升级为“首次登录一次，后续默认复用同一套 Ordo 托管浏览器会话”，并且在会话临近失效或已经失效时给出明确提醒。

## 环境前提

- 系统已安装 Chrome 或 Chromium
- 仓库根目录存在 `publish.py`
- 当前版本包含浏览器托管会话改造

## 默认落点

- 托管 profile：`.tiandidistribute/browser-session/profile`
- 会话状态：`.tiandidistribute/browser-session/state.json`
- 默认调试端口：`9333`

## 本轮收口结果

- 已确认此前无法稳定落到托管端口，并不是配置关闭，而是两层原因叠加：
  - macOS 托管启动仍使用 `open -a`，在系统 Chrome 已运行时无法可靠拉起独立实例
  - `ensure_chrome_ready()` 只要看到现有系统标签页就直接复用，导致根本不会尝试切换到托管来源
- 本轮已修复：
  - macOS 托管启动改为 `open -na`
  - 当 `browser_session.enabled=true` 且当前 CDP 来源不是 `managed_browser_*` 时，`ensure_chrome_ready()` 会强制继续拉起托管实例，而不是被系统标签页短路
- 隔离实测结果：
  - 通过临时目录 + 临时端口执行正式代码路径 `ensure_chrome_ready()`
  - 返回来源为 `managed_browser_port`
  - 托管 profile 目录已创建
  - 说明“独立 profile + 独立调试端口 + 来源判定”这一整条链路已经打通
- 主工作区真实验收结果：
  - 已在主工作区成功切到 `managed_browser_port 9333`
  - 已在托管浏览器中完成知乎、头条号、一点号、简书登录
  - 首次登录后预检可通过，四个平台均可绑定到托管工作台 target
  - 关闭托管实例后重新拉起，`managed_browser_port` 来源仍能恢复
  - 经过本轮收口后，托管实例重启后会自动恢复 / 补齐知乎、头条号、一点号、简书四个平台标签页
  - 预检在页面刚恢复时可能短暂遇到单个平台编辑器尚未挂载完成，短等待后可恢复为 `blockers=[]`
  - 因而当前可以确认：托管会话复用已成立，且“重启后自动补齐平台标签页”也已收口

## 验证步骤

在执行本页验证时，建议同时参考 `docs/manual-validation/2026-03-28-browser-smoke.md`，因为本轮不仅要确认会话可复用，还要在真实页面确认 `封面 / AI 声明` 的任务级开关是否生效。

### 1. 首次登录

1. 启动桌面工作台，或执行一次浏览器平台预检 / 发布。
2. 确认 Ordo 会优先拉起自己的托管浏览器会话。
3. 在这套浏览器里分别登录知乎、头条号、一点号（如需要也可补简书）。
4. 关闭工作台与浏览器。

预期结果：

- 浏览器资料写入托管 profile，而不是依赖临时 Chrome 会话。
- 顶部状态区显示“浏览器会话已托管”。

### 2. 重启后复用

1. 重新启动工作台。
2. 直接进入浏览器平台相关预检或草稿发布。

预期结果：

- 不需要重新授权。
- `publish.py` 优先连接托管调试端口，而不是要求手动先开远程调试 Chrome。
- `state.json` 中对应平台会写入 `last_checked_at` / `last_healthy_at`。
- 工作台顶部状态区应继续显示“浏览器会话已托管”，而不是回退到系统 Chrome。
- 在 macOS 上，即使系统 Chrome 已经打开，Ordo 也应继续拉起独立托管实例，而不是静默复用系统实例。

### 3. 临近失效提醒

1. 手动把 `.tiandidistribute/browser-session/state.json` 中某个平台的 `last_healthy_at` 改成早于提醒阈值很多天。
2. 重新打开工作台并刷新资源。

预期结果：

- 顶部状态区显示“浏览器会话临近失效”。
- 执行区资源提示出现“建议重新打开托管浏览器确认登录状态”。
- 对应平台会写入 `last_reminded_at`。

### 4. 失效 / 重新登录提醒

1. 在托管浏览器中主动退出某个平台，或让页面停留在登录页。
2. 再次执行浏览器平台预检。

预期结果：

- 预检阻塞并明确指出当前平台需要重新登录。
- 顶部状态区切换为“浏览器会话需重登”。
- `state.json` 中对应平台写入 `status=expired_or_relogin_required` 与 `last_relogin_required_at`。

## 与发布选项联动的补充检查

当会话复用通过后，建议立刻在工作台里补三组最小回归：

1. `封面: 自动`、`AI 声明: 自动`
2. `封面: 关闭`、`AI 声明: 关闭`
3. `封面: 强制开启`、`AI 声明: 强制开启`

预期：

- 工作台重新规划任务后，执行区提示会同步显示当前选项状态。
- `force_off` 时，知乎 / 头条号 / 一点号应跳过对应声明；封面策略应按平台能力处理。
- 一点号在直接发布模式下若无法彻底无封面，应明确回退为默认封面，而不是静默失败。

## 回退说明

- 如果托管浏览器实例不可用，系统会暂时回退到系统 Chrome / `DevToolsActivePort` 探测链路，避免第一阶段直接把原有能力打断。
- 一旦发生回退，桌面工作台应明确显示当前不是完全托管模式。
- 若再次观察到来源长期停留在 `macos_devtools_port_file`，优先检查两点：托管 profile 是否被创建、以及本轮启动是否真的走到了 `open -na ... --remote-debugging-port=<port>`。
