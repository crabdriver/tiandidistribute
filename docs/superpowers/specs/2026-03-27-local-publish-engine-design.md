# 本地多平台发布引擎设计

日期：2026-03-27

## 当前实现状态说明

本文是 Ordo 本地发布引擎的阶段设计文档。当前仓库已经在该设计基础上继续推进，现状可概括为：

- 第一阶段的 `GUI-ready` 引擎核心、工作台桥接与结构化结果输出已经基本落地
- 仓库中已经存在 `desktop/` 下的桌面工作台 MVP，而不是只有 CLI 入口
- 但“恢复/续跑完整产品化、安装分发、跨平台打包”仍未完成，依然属于后续阶段目标

因此，阅读本文时应把它理解为“架构目标与阶段边界说明”，而不是当前代码状态的逐项事实清单。

## 1. 背景

`ordo` 当前已经具备以下基础能力：

- 以 Markdown 为单一内容源的多平台分发流程
- 微信公众号排版、主题、预览、封面和草稿发布能力
- 知乎、头条号、简书、一点号的浏览器自动化发布能力
- 基于发布控制台的人工确认式批量执行雏形
- AI 生图与评论自动回复等辅助能力

但当前项目仍主要表现为一组可运行脚本的集合，而不是一个清晰、稳定、可被图形界面直接调用的本地发布引擎。

本设计的目标，是先把 `ordo` 整理成一个 `GUI-ready` 的本地核心引擎，再在后续阶段接入原生桌面 GUI，并最终打包为适用于 `macOS` 和 `Windows` 的一键安装软件。

需要额外说明的是：第一阶段聚焦“发布主链路引擎化”。现有 AI 生图、评论自动回复等辅助能力在这一阶段默认继续作为独立入口保留，不要求并入同一任务模型，除非它们直接参与发布主链路的必要步骤。

对外发布时，项目正式名称统一为 `ordo`。为兼容现有工作流与本地状态目录，仓库中的部分内部路径和缓存目录暂时仍保留 `.tiandidistribute/` 命名，这不影响后续逐步迁移。

## 2. 产品定位

本项目的第一性定位调整为：

- 面向中文内容创作者的本地多平台自动发布引擎
- 未来会被封装为原生桌面软件
- 当前阶段优先整理核心发布能力，而不是优先做 GUI 外壳

它不再只是一个本地 CLI 工具集合，而是未来桌面产品的核心执行层。

## 3. 自动化边界

当前阶段定义的“全自动多平台分发”边界如下：

- 前提：用户本机已经具备可用登录态
- 系统负责：文章装载、内容转换、平台注入、草稿或发布、结果记录、失败恢复
- 系统不负责：自动登录、验证码处理、风控绕过、云端托管执行

也就是说，第一阶段追求的是“登录态已存在条件下的稳定自动执行”，而不是覆盖账号体系和平台风控的全部复杂性。

## 4. 第一阶段目标

第一阶段的交付物应满足以下目标：

- 形成一个可独立运行的本地发布引擎
- `CLI` 只是核心引擎的薄入口
- 未来 `GUI` 可以直接调用同一套能力
- 支持批量任务、状态跟踪、错误分类、恢复执行、结果汇总

第一阶段完成后，项目应该从“脚本集合”转变为“具备统一任务模型和状态模型的本地执行内核”。

## 5. 核心模块划分

建议将第一阶段的核心引擎拆分为 5 个模块。

### 5.1 任务层

任务层负责接收外部输入并生成统一任务对象，包括：

- 单篇发布任务
- 目录批量发布任务
- 任务恢复与重试任务

该层需要统一表达以下信息：

- 文章列表
- 目标平台列表
- 发布模式（草稿或正式发布）
- 主题和封面策略
- 恢复点和任务 ID

任务层同时承担运行时调度编排职责，也就是负责决定任务如何按“文章 x 平台”的粒度推进；本文后续提到的“调度层”默认归属于任务层内部，而不是额外单列一个第六模块。

### 5.2 配置层

配置层负责统一整理当前分散的配置来源，包括：

- `secrets.env`
- `config.json`
- 命令参数
- 将来 GUI 写入的本地配置

该层需要给上层提供一致的配置读取模型，避免同一类信息在多个脚本中使用不同字段和优先级。

### 5.3 平台适配层

每个平台保留各自的发布细节，但统一实现相同的适配边界。每个平台适配器至少需要支持：

- `prepare`
- `publish`
- `verify`
- `collect_result`

这样调度层只依赖统一契约，而不依赖具体平台页面细节。

### 5.4 状态与恢复层

状态与恢复层是第一阶段的核心资产，负责：

- 跟踪任务级、文章级、平台级、步骤级状态
- 记录失败原因、可重试性和恢复点
- 支持中断后继续、跳过已成功项、重试失败项

当前 `publish_console_state` 可以视为该层的早期雏形，但需要被提升为整个引擎的统一状态模型，而不是只服务发布控制台。

### 5.5 日志与结果层

日志与结果层负责输出结构化执行结果，而不是依赖控制台文案来判断执行状态。

该层至少需要提供：

- 统一状态码
- 错误类别
- 原始输出
- 诊断摘要
- 汇总报表

未来 CLI 可以将其打印为文本，GUI 可以将其直接渲染为任务面板和错误视图。

## 6. 统一数据流

一条发布任务应当走统一的数据流，而不再依赖松散的脚本互调。

建议的数据流如下：

1. 接收任务
2. 标准化任务对象
3. 配置解析与预检
4. 文章准备
5. 平台执行
6. 状态落盘
7. 任务收尾与汇总

### 6.1 接收任务

外部入口提交任务，输入可能包括：

- Markdown 文件或目录
- 目标平台
- 发布模式
- 主题策略
- 是否从历史任务恢复

### 6.2 标准化任务对象

系统将原始输入整理为统一任务模型，例如：

- 本次任务包含多少篇文章
- 每篇文章要发布到哪些平台
- 每个平台的模式和上下文
- 当前任务唯一 ID

### 6.3 配置解析与预检

统一做配置校验和执行条件校验，包括：

- 必要配置是否存在
- 浏览器连接是否可用
- 登录态是否可用
- 平台是否具备基础执行条件

应尽量把确定会失败的问题前置暴露，而不是在执行中途才报错。

### 6.4 文章准备

将 Markdown 转换为平台可发布的内容：

- 微信链路负责微信兼容 HTML、封面策略和草稿材料
- 浏览器平台负责标题提取、正文转换和平台特定内容准备

### 6.5 平台执行

任务层中的调度编排按“文章 x 平台”的粒度推进。每个平台执行时，至少经历 `prepare -> publish -> verify -> collect_result` 这条统一链路；其中 `verify` 用于发布后校验实际结果，而不是独立于执行链路之外的额外阶段。每一步都返回结构化结果，而不是只依赖打印日志。

### 6.6 状态落盘

每完成一个平台或一篇文章，都持久化最新状态，以便：

- 程序中断后恢复
- GUI 实时读取进度
- 调试和问题诊断

### 6.7 任务收尾

输出统一汇总信息，例如：

- 成功数量
- 失败数量
- 平台级失败分布
- 待重试项

## 7. 错误处理与恢复策略

第一阶段的重点不是“永不失败”，而是“失败时能正确识别、分类并继续推进”。

### 7.1 错误分类

建议至少区分以下错误类型：

- `config_error`
- `environment_error`
- `login_required`
- `platform_changed`
- `content_rejected`
- `rate_limited`
- `transient_error`
- `duplicate_or_skipped`
- `unknown_error`

### 7.2 恢复策略

不同错误类型应绑定不同默认动作：

- `config_error`：阻塞整个任务
- `environment_error`：阻塞任务并提示修复本机环境
- `login_required`：暂停相关平台并等待用户恢复登录态
- `platform_changed`：停止该平台后续自动执行
- `content_rejected`：跳过当前文章当前平台
- `rate_limited`：停止该平台当日后续任务；“当日”默认按本机自然日处理，后续如有必要可配置
- `transient_error`：自动重试若干次
- `duplicate_or_skipped`：记为已处理

### 7.3 重试边界

自动重试只建议用于临时性错误，例如：

- CDP 超时
- 页面短暂未加载完成
- 网络请求失败
- 临时上传失败

以下情况不建议盲目自动重试：

- 登录失效
- 页面元素缺失
- 平台规则拒绝
- 标题或内容不合规
- 发布次数上限

### 7.4 恢复粒度

恢复能力应覆盖：

- 任务级
- 文章级
- 平台级
- 步骤级

恢复时应默认跳过已成功项，从失败点或未执行项继续。

### 7.5 诊断信息

每次失败至少应记录：

- 平台
- 当前步骤
- 错误类型
- 错误摘要
- 原始 stdout / stderr
- 当前页面 URL
- 时间戳
- 是否可重试

如果后续需要增强 GUI 诊断能力，可以继续补充截图或页面快照路径。

## 8. 测试与验证边界

第一阶段不需要证明“所有平台永远稳定”，但必须证明核心引擎具备可验证性和可回归性。

### 8.1 核心单元测试

重点覆盖纯逻辑能力：

- 任务对象生成
- 配置解析与优先级
- 状态流转
- 错误分类
- 恢复决策
- 结果汇总

### 8.2 平台适配契约测试

验证每个平台是否遵守统一接口与返回模型，而不依赖真实网站。

### 8.3 调度与恢复集成测试

重点覆盖以下场景：

- 多篇文章批量执行
- 中途失败后恢复
- 跳过已成功项
- 平台级重试
- 任务中断后继续
- 部分平台不可用时的行为

### 8.4 人工验证矩阵

由于平台自动化依赖真实网页，仍需要人工验证清单来确认平台是否改版或行为是否变化。至少包括：

- 能否进入编辑页
- 标题能否写入
- 正文能否注入
- 草稿或发布按钮能否触发
- 结果是否被正确识别
- 失败诊断是否充分

## 9. 第一阶段明确不做的事

为避免范围失控，第一阶段明确不做以下事项：

- 完整 GUI 产品界面
- 远程服务化与云端执行
- 自动登录和风控处理
- 所有异常自动修复
- 扩展更多平台；第一阶段只整理现有平台的统一契约和回归能力
- 复杂插件系统
- 一开始就做完整跨平台产品化细节打磨

第一阶段只聚焦于“GUI-ready 的本地发布引擎”。

## 10. 推荐阶段路线

建议按 3 个阶段推进：

### 阶段一：整理本地发布引擎

目标：

- 从脚本集合整理成可调用核心引擎
- 统一任务、配置、状态、结果和错误模型

### 阶段二：接入桌面 GUI 外壳

目标：

- 使用 `Tauri + Rust` 路线接入原生桌面 GUI
- 将任务管理、配置、日志和状态展示图形化

### 阶段三：安装与跨平台分发

目标：

- 面向 `macOS` 和 `Windows` 完成打包和分发
- 整理安装、配置落盘、浏览器依赖和升级体验

## 11. 当前结论

当前阶段最重要的工作，不是继续零散修补单个脚本，而是先把项目从“脚本集合”重组为“可被 GUI 调用的本地引擎”。

在这一方向下，最优先的不是新界面，而是统一：

- 任务模型
- 配置模型
- 状态模型
- 错误模型
- 平台适配契约

只有这些基础收敛之后，后续 GUI、安装器和跨平台产品化工作才不会建立在不稳定的脚本边界之上。

## 12. 外部项目借鉴与功能映射

在后续设计与产品化过程中，`ordo` 可以明确借鉴两类外部项目经验：

- `xiaohu-wechat-format`：更适合借鉴主题体系、可视化画廊、封面模块、内容增强和配置组织方式
- `mopai-markdown`：更适合借鉴 GUI 工作台、导入体验、预览体验、设置面板和整体产品交互

两者在 `ordo` 中的推荐映射方式不是整套照搬，而是：

- 底层能力更接近 `xiaohu-wechat-format`
- 上层体验更接近 `mopai-markdown`

### 12.1 内容导入层

`ordo` 后续应支持以下内容入口：

- 粘贴单篇文章
- 导入单文件：`Markdown`、`TXT`、`DOCX`
- 导入文件夹

导入后的目标不是保留原始富文本排版，而是统一转成干净的内部文章模型，重点保留：

- 标题层级
- 段落
- 列表
- 图片资源
- 来源路径和基础元数据

这一层更适合借鉴 `mopai-markdown` 的 GUI 导入体验，而不是直接借其前端实现。

### 12.2 模板与主题系统

`ordo` 的模板系统建议吸收 `xiaohu-wechat-format` 的主题分层思路，并结合 GUI 工作台做产品化表达。推荐能力包括：

- 主题分类
- 默认随机主题
- 自定义逐篇主题选择
- 后续演进为“布局模板 + 配色模板”的组合体系
- 可视化主题预览和主题画廊

其中，当前已确认的批量发布模板流程为：

- 批量导入后先弹出一次全局模板选择
- `默认模板`：系统为每篇文章随机分配主题
- `自定义模板`：进入逐篇配置，允许用户为每篇文章指定主题和样式

### 12.3 封面系统

封面能力不应只服务微信平台，而应扩展成平台级策略模块。

当前已确认的非微信平台封面策略为：

- 适用平台：知乎、头条号、简书、一点号
- 封面来源：默认封面池
- 分配粒度：`文章 x 平台`
- 默认行为：同一篇文章在不同非微信平台可以随机分配不同封面

封面池本身不应继续写死为 7 张，而应升级为可扩展资源目录，后续支持扩充到十几张，并尽量避免短周期重复。

### 12.4 预览层

`ordo` 后续应同时吸收两类预览能力：

- 来自 `xiaohu-wechat-format` 的真实文章主题预览和画廊比较
- 来自 `mopai-markdown` 的工作台预览和手机视图体验

建议至少具备：

- 当前文章主题预览
- 封面预览
- 手机视图预览
- 平台差异预览

### 12.5 设置中心

`ordo` 后续应形成独立设置中心，而不是长期依赖命令参数堆叠。建议收纳：

- 默认模板模式
- 默认主题池
- 默认封面池
- 导入目录
- 浏览器工作台
- 发布模式
- 自动保存策略

## 13. GUI 工作台方向补充

结合已确认需求，`ordo` 的 GUI 主界面方向应明确为“工作台式”，而不是单纯向导页。

推荐的信息架构如下：

- 左侧：文章列表与导入入口
- 中间：全局模板选择、当前文章的主题与样式配置、正文预览
- 右侧：非微信平台封面池、当前文章各平台封面分配结果、平台设置
- 底部或侧边：发布日志、错误摘要、执行状态

### 13.1 批量发布前流程

推荐流程为：

1. 导入文章内容
2. 生成文章列表
3. 弹出一次全局模板选择
4. 若为默认模板，则为每篇文章随机分配主题
5. 若为自定义模板，则逐篇指定主题和样式
6. 为非微信平台分配默认封面
7. 进入正式发布阶段

### 13.2 默认与自定义模式

为了兼顾效率和控制力，后续 GUI 需同时支持两种发布前模式：

- `默认模板`：批量高效，适合快速发布
- `自定义模板`：逐篇可控，适合重点文章

系统在任何情况下都应具备合理兜底能力：

- 即使用户不自定义主题
- 即使用户不手动指定封面
- 系统也能自动完成主题和封面的分配，再进入发布执行

## 14. GUI 一期最小可用范围

在新增需求确认后，`ordo` 的 GUI 一期范围进一步收敛为 5 个核心模块：

1. 内容导入
2. 批量文章列表
3. 模板与主题
4. 封面中心
5. 发布助手

### 14.1 内容导入

支持以下入口：

- 粘贴单篇文章
- 导入单文件：`Markdown`、`TXT`、`DOCX`
- 导入文件夹

导入后的内容统一转换为内部文章模型，不追求保留原始富文本排版，而是保留标题、段落、列表、图片和基本元数据。

### 14.2 批量文章列表

文章列表负责承载批量任务的主工作流，至少应显示：

- 文章标题
- 来源
- 当前模板模式
- 当前主题
- 当前封面状态
- 发布状态

### 14.3 模板与主题

这一模块负责：

- 全局模板模式选择
- 默认主题随机分配
- 自定义逐篇主题指定
- 当前文章样式确认

### 14.4 封面中心

这一模块负责：

- 非微信平台默认封面池管理
- 当前文章在知乎、头条、简书、一点号上的封面分配结果
- 后续扩展手动覆盖能力

### 14.5 发布助手

这一模块负责：

- 平台勾选
- 发布前确认
- 执行日志
- 错误提示
- 后续重试与恢复入口

### 14.6 一期明确后放的能力

以下能力属于后续增强，不纳入 GUI 一期硬目标：

- 完整手机预览
- 复杂主题画廊比较
- 封面历史去重策略面板
- 完整富文本编辑器
- 云端同步与账号托管

## 15. 工作台首页交互顺序

`ordo` 的 GUI 首页本质上不是编辑器首页，而是批量发布工作台首页。推荐交互顺序如下：

1. 导入内容
2. 生成批量文章列表
3. 弹出一次全局模板选择
4. 完成主题与封面分配
5. 发布前复核
6. 进入正式发布
7. 保留结果回看

### 15.1 初始状态

首页默认呈现“空工作台”状态，重点显示：

- 粘贴单篇文章
- 导入单文件
- 导入文件夹
- 最近一次草稿或任务

### 15.2 导入后的工作状态

导入完成后，工作台切换为三栏结构：

- 左侧：文章列表
- 中间：模板与当前文章配置
- 右侧：封面与平台设置

### 15.3 全局模板选择

全局模板选择只在导入后触发一次：

- `默认模板`
- `自定义模板`

选中 `默认模板` 后，系统立即为每篇文章随机分配主题；选中 `自定义模板` 后，进入逐篇配置工作流。

### 15.4 发布前复核

在进入执行前，工作台应提供统一复核视图，展示：

- 待发布文章数
- 每篇文章的当前主题
- 每个平台的封面状态
- 待发布平台
- 未完成配置项

### 15.5 发布执行与结果回看

进入执行后，工作台重心转为状态和日志：

- 当前正在发布的文章
- 当前平台
- 成功/失败/跳过状态
- 错误摘要
- 结果回看与后续重试入口

## 16. GUI 模块与底层对象映射

为了避免 GUI 直接操作脚本，界面层应建立在稳定的数据对象之上。建议至少定义以下核心对象：

### 16.1 `ImportJob`

用于表示一次导入动作，包含：

- 导入方式
- 来源路径或原始文本
- 导入时间
- 导入结果
- 生成文章数量

### 16.2 `ArticleDraft`

用于表示系统中的单篇文章对象，包含：

- 标题
- 正文
- 图片资源
- 来源路径
- 字数
- 当前模板模式
- 当前主题
- 当前配置完成状态

### 16.3 `TemplateAssignment`

用于表示文章的模板分配结果，包含：

- 模板模式
- 当前主题 ID
- 是否随机生成
- 是否人工覆盖
- 是否已确认

### 16.4 `CoverAssignment`

用于表示平台级封面分配结果，包含：

- 文章 ID
- 平台 ID
- 封面路径或资源 ID
- 封面来源类型
- 是否随机分配
- 是否人工覆盖

### 16.5 `PublishJob`

用于表示一次真实批量发布任务，包含：

- 本次任务的文章列表
- 目标平台
- 当前任务状态
- 当前执行位置
- 成功/失败/跳过统计
- 错误详情
- 是否可恢复

### 16.6 对象关系

推荐关系如下：

- `ImportJob` 生成多个 `ArticleDraft`
- 每个 `ArticleDraft` 对应一个 `TemplateAssignment`
- 每个 `ArticleDraft` 对应多个 `CoverAssignment`
- 用户点击开始发布后，系统将这些对象组合成一个 `PublishJob`

## 17. GUI 一期页面与面板组件拆分

为了让 GUI 一期可以尽快落地，页面结构建议保持克制，优先围绕单一主工作台展开。

### 17.1 主页面：发布工作台

这是一期最核心的页面，承担从导入到执行的完整链路。推荐拆成以下区域：

- 顶部工具条：新建导入、最近任务、平台筛选、开始发布
- 左侧文章列表：文章卡片、批量选择、搜索、状态筛选
- 中间主配置区：模板模式、主题选择、正文预览、当前文章摘要
- 右侧侧栏：封面池、当前文章平台封面分配、平台开关和发布设置
- 底部执行区：日志流、结果统计、失败重试入口

### 17.2 弹层：导入面板

导入相关交互建议用弹层或抽屉承载，避免切走主工作台。至少包含：

- 粘贴文章输入区
- 单文件选择
- 文件夹选择
- 导入预检查提示
- 导入结果摘要

### 17.3 弹层：全局模板选择

批量导入完成后立即弹出，作用是决定整批任务走：

- `默认模板`
- `自定义模板`

如果选择 `默认模板`，系统应立即完成主题分配并返回工作台；如果选择 `自定义模板`，系统应默认停留在第一篇文章的配置面板。

### 17.4 面板：逐篇文章配置

在选中某篇文章后，主配置区应切换为当前文章编辑视角，至少显示：

- 当前文章标题和来源
- 当前主题
- 当前模板模式
- 当前封面分配结果
- 当前平台勾选状态
- 缺失配置提示

### 17.5 面板：封面分配区

封面面板是右侧侧栏的重点，至少包含：

- 默认封面池缩略图
- 当前文章在知乎、头条、简书、一点号上的封面映射
- 随机分配结果
- 后续手动替换入口预留位

### 17.6 面板：发布执行区

执行区不只是打印日志，还应具备任务面板属性，至少显示：

- 当前任务 ID
- 当前文章 / 当前平台
- 已完成 / 失败 / 跳过统计
- 最新错误摘要
- 可恢复状态提示

### 17.7 二期再补的页面

以下页面可以在一期后补，不要求和首版一起上线：

- 独立设置中心
- 完整预览页
- 主题画廊页
- 封面历史与素材管理页
